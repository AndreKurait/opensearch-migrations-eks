## Why Migration Assistant?

Native snapshot/restore only works within one major version gap and requires downtime. The `_reindex` API reads every document through the source cluster's HTTP layer — slow, fragile, and puts heavy load on production. **Reindex-from-Snapshot (RFS)** takes a fundamentally different approach:

<CardGrid>
  <Card title="Zero Source Load" icon="approve-check-circle">
    Takes a one-time snapshot, then reads raw Lucene segment files directly from S3. After the snapshot, the source cluster is never touched again.
  </Card>
  <Card title="Any Version Gap" icon="random">
    Works across any supported version gap (e.g., ES 1.x → OS 3.x) — no intermediate upgrades required.
  </Card>
  <Card title="Massive Parallelism" icon="up-caret">
    One worker per shard, all reading from S3 independently. Peak throughput: 590K docs/min per 2 vCPU worker.
  </Card>
  <Card title="Full Resumability" icon="rocket">
    Failed shards are automatically retried. Completed shards are never reprocessed. Work is never lost.
  </Card>
</CardGrid>

## Migration Scenarios

<CardGrid>
  <Card title="Backfill Only" icon="open-book">
    Snapshot → Metadata → RFS backfill → Verify → Switch traffic. Best when you can tolerate a brief write pause.
  </Card>
  <Card title="Capture & Replay" icon="random">
    Deploy proxy fleet → Record to Kafka → Replay on target. Validates target behavior with real production traffic.
  </Card>
  <Card title="Combined (Zero-Downtime)" icon="star">
    Start capture first → Snapshot → Metadata → Backfill → Replay catches up → Verify → Switch. The complete path.
  </Card>
</CardGrid>
