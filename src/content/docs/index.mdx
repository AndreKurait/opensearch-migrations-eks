## How It Works

<Steps>
1. **Deploy** (~10 min) — Run the bootstrap script to create an EKS cluster with Migration Assistant

2. **Configure** (~5 min) — Define source, target, and migration options in YAML from the Migration Console

3. **Migrate** (varies by data size) — Submit the workflow. It snapshots, migrates metadata, backfills documents, and pauses for your approval at each stage

4. **Validate & Switch** (~15 min) — Verify document counts and query results, then redirect traffic to the target
</Steps>

## Why Migration Assistant

<CardGrid>
  <Card title="Zero Source Impact" icon="approve-check-circle">
    Reads raw Lucene files from a snapshot in S3 — your source cluster is never touched after the initial snapshot. No performance hit.
  </Card>
  <Card title="Skip Version Gaps" icon="random">
    Go directly from Elasticsearch 1.x to OpenSearch 3.x. No stepping through 5.x → 6.x → 7.x → OS 1.x. One migration, any gap.
  </Card>
  <Card title="Scale Independently" icon="up-caret">
    Each worker processes one shard from S3. Add workers to go faster — **590K docs/min** per worker. Failed shards retry automatically.
  </Card>
  <Card title="Declarative Workflows" icon="puzzle">
    Define your migration as YAML. Argo Workflows handles parallel execution, retry logic, and approval gates before critical steps.
  </Card>
</CardGrid>

## Before You Start

:::note[Prerequisites]
- **AWS account** with permissions to create EKS clusters, S3 buckets, and IAM roles
- **kubectl** and **Helm 3** installed locally
- **Source cluster** accessible from your VPC (Elasticsearch 1.x–8.x or OpenSearch 1.x–2.x)
- A **snapshot** in S3 — or the ability to create one
:::
